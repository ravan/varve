# Varve v1 benchmark report

Machine: Apple M3 Max (14-core CPU variant, per `sysctl hw.ncpu`), 36 GB RAM, macOS 26.3
(Darwin 25.3.0). Rust 1.93.0 (254b59607, 2026-01-19). Release builds (`cargo run --release`,
`cargo bench`, `cargo test --release`).

Date: 2026-07-12. Commit: `fd37a92` (HEAD at measurement time, before this report was
committed). Reproduce: the exact command is listed at the top of each section below; every
number in this report was captured by running that command on this machine during this
session (nothing here is carried over from `STATUS.md` or a prior run).

**Machine load during measurement** (recorded via `uptime` / `df -h .`, a single snapshot taken
at the start of the run — not a before/after comparison): load average 4.71 / 5.63 / 7.28
(1/5/15-min) on a 14-core box — the box was under moderate shared load (other processes were
competing for CPU) — and 95 GiB free on the benchmark disk (`/dev/disk3s5`, 90% used overall).
This is a laptop under everyday load, not an isolated/quiet benchmark rig — treat absolute
numbers as indicative of the right order of magnitude, not lab-grade precision. Docker was
available (`docker ps` worked, 0 containers running) but no MinIO/S3 container or compose file
exists in this worktree; S3/object-store numbers below are marked "not measured" rather than
invented (see Honest notes).

## Spec §13 targets vs measured

Targets are quoted verbatim from `docs/design/2026-07-04-varve-design.md` §13:
"**Performance targets (v1, laptop = M-series/16 GB):** ≥10k write ops/s sustained (batched),
warm point lookup < 1 ms, warm 2-hop over 1M-node graph < 50 ms, AS-OF historical query within
2× of current-time equivalent. Server: ≥5k tx/s on object-store log via group commit; query
throughput scales ~linearly to 4 nodes on the read benchmark. Targets, not guarantees — tracked
in the benchmark report."

| Target (spec §13, laptop) | Measured | Verdict | Source |
|---|---|---|---|
| ≥10k write ops/s sustained (batched) | 41,545 events/s bulk batched `INSERT` (1,000 rows/statement, 1 tx/statement, no GQL `MATCH`) | **met** | `block_bench` |
| warm point lookup < 1 ms | 0.38 ms (p50 and avg of 100 warm in-process reads) | **met** | `social_bench` |
| warm 2-hop over 1M-node graph < 50 ms | 18.70 ms warm avg (18.59 ms p50) — measured on the 10k-node/60k-edge fixture; the 1M-node graph was **not ingested** this run | **not measured** at 1M-node scale (comfortably passes on the smaller fixture — see Honest notes) | `traversal_bench` / `social_bench` |
| AS-OF historical within 2× of current | 0.94× (17.15 ms AS-OF-late-anchor p50 vs 18.20 ms current-time p50 — historical was actually *faster*) | **met** | `social_bench` |
| server ≥5k tx/s on object-store log (group commit) | not measured — no MinIO/S3 backend stood up this run; the only comparable local number is write_bench's single-statement local-fsync group-commit path at 372 tx/s, which is a *different regime* (embedded single-writer, local disk, one row per tx, no object-store round trip) | **not measured** (optional this run; local proxy is far below and not a substitute — see Honest notes) | `write_bench` (local-fsync proxy only) / `cache_bench` (S3 path skipped, no docker MinIO) |

## Write path

Reproduce:
```
cargo run --release --example write_bench -p varve
cargo run --release --example block_bench -p varve
cargo run --release --example cache_bench -p varve
```

**`write_bench`** — single-row `INSERT` per tx, 8 concurrent workers, 4,000 txs total
(unbatched — this is the fsync-bound floor, not the batched target):
```
        memory: 4000 txs / 8 workers →     7038 tx/s  (568.33ms)
 local (fsync): 4000 txs / 8 workers →      372 tx/s  (10.76s)
```
This is healthy — the write path is not degraded. The 372 tx/s local-fsync number is
the **single-statement, fsync-bound tx/s** figure: one GQL statement = one row = one committed
tx with a real fsync inside a 15 ms group-commit window. It is intentionally *not* the number
used to judge the ≥10k-write-ops/s target, because that target is qualified "(batched)".

**`block_bench`** — bulk ingest of 1,000,000 single-node events via multi-row `INSERT`
statements (1,000 rows/statement, `max_block_rows = 25,000`), no `MATCH`, no GQL edge
resolution — this is the **batched bulk path** the ≥10k-write-ops/s target is measured
against:
```
ingest: 1000000 events in 24.07s (41545 events/s)
blocks flushed: 39 persisted block files under store/v1/blocks
reopen (manifest + log tail): 48.36ms
point lookup: cold 21.22ms, warm 7.55ms
exit criterion (<100ms warm point lookup): PASS
full scan after restart: 1000000 rows OK
```
41,545 events/s is >4× the ≥10k target. Note the "warm" 7.55 ms point-lookup figure here
includes a fresh `Db::open()` (manifest + log-tail replay) before the lookup — see Honest
notes on why this number is not directly comparable to `social_bench`'s 0.38 ms warm figure.

**`cache_bench`** — 100,000-event ingest (500 rows/statement) on local FS with the disk cache
enabled, then cold vs warm point lookup across a process restart (S3/MinIO path not run — see
Honest notes):
```
ingest  100000 events (200 txs): 3.636515125s
cold    (disk cache empty):     open+lookup 18.935041ms
warm    (cache survived reopen): open+lookup 7.748834ms
disk cache: 12 entries under <tmp>/cache
```
≈27,500 events/s on this batching profile (200 txs × 500 rows / 3.64 s), consistent with
`block_bench`'s bulk-batched throughput order of magnitude.

## Read path

Reproduce:
```
cargo run --release --example traversal_bench -p varve
cargo run --release --example social_bench -p varve
```

Both run the same deterministic 10k-node/60k-edge social fixture
(`varve_testkit::fixture::social_graph`, seed 42) on a durable local (log + block-store)
profile.

**`traversal_bench`**:
```
ingest 610 programs (10000 nodes, 60000 edges) in 101.81s (6 programs/s)
reopen (manifest + log tail): 10.65ms
ingest 101.81s · 6 programs/s · 2-hop (33 rows) cold 22.00ms warm avg 18.70ms p50 18.59ms · {1,3} (211 rows) cold 33.28ms warm avg 29.43ms p50 29.20ms
exit criterion (warm 2-hop < 50ms): PASS
```

**`social_bench`** (adds a warm point-read figure and the AS-OF/current ratio):
```
ingest 610 txs (10000 nodes, 60000 edges = 70000 events) in 101.78s (688 events/s, 6 tx/s)
reopen (manifest + log tail): 11.64ms
point read (1 row) cold 975.79µs warm avg 381.46µs p50 377.67µs
2-hop current (33 rows) cold 20.30ms warm avg 18.22ms p50 18.20ms
2-hop AS-OF mid anchor (0 rows, 0 edges exist yet) cold 12.94ms warm avg 12.49ms p50 12.43ms — not the headline AS-OF number
2-hop AS-OF late anchor (15 rows, ~half the edges landed) cold 16.95ms warm avg 17.20ms p50 17.15ms

| metric | value |
|---|---|
| ingest (batched) | 688 events/s (610 txs, 101.8 s) |
| warm point read (p50/avg of 100) | 0.38 ms / 0.38 ms |
| warm 2-hop (p50/avg of 100) | 18.20 ms / 18.22 ms |
| AS-OF historical 2-hop (p50) | 17.15 ms (0.94x of current) |
```

Warm point read (0.38 ms) and warm 2-hop (18.22 ms avg) are both comfortably inside their
targets (<1 ms, <50 ms) **on this 10k/60k fixture**. The AS-OF/current ratio is 0.94×, i.e.
the historical query was *faster* than the current-time one this run (fewer edges existed at
the AS-OF anchor, so less to traverse) — well inside the ≤2× target either way.

## Scale-out

Reproduce: `just bench-scale-out` (env-gated: `VARVE_SCALE_BENCH=1 cargo test -p varve-server
--release --test scale_out_bench -- --nocapture --test-threads=1`).

```
| query nodes | aggregate reads | window | QPS |
|---|---|---|---|
| 1 | 1522 | 10s | 152 |
| 2 | 2825 | 10s | 282 |
| 4 | 5305 | 10s | 530 |
```

Scaling factor: 1.86× at 2 nodes, 3.49× at 4 nodes (relative to the 1-node baseline) — near
linear (87% efficiency at 4×), consistent with "read throughput ~linear to 4 nodes." **Caveat:
all four "query nodes" in this bench are processes on the single machine described in the
header above, sharing the same CPU cores and disk** — this demonstrates the read-scale-out
*mechanism* works and the slope is encouraging, but it is not a measurement of network-attached,
separate-hardware scale-out. Treat it as indicative, not a datacenter capacity claim; the
target should be re-measured on real distributed hardware (separate machines/VMs, real network
hops to a shared object store) before being cited as verified.

## Honest notes

- **1M-node 2-hop target not measured.** The fixture used by both `traversal_bench` and
  `social_bench` is 10,000 nodes / 60,000 edges, not 1,000,000 nodes. The measured warm 2-hop
  on that fixture is 18.70 ms (well under the 50 ms target), but this run does **not** claim
  the 1M-node target as met or verified — it was not exercised, and 2-hop latency at 100×
  fixture scale should not be silently extrapolated (fan-out, block count, cache pressure, and
  trie depth all change nonlinearly with graph size). Closing this requires ingesting a
  1M-node/~6M-edge graph with the same fixture generator and re-running `traversal_bench` /
  `social_bench` against it — a multi-hour ingest at the ~590 edges/s GQL edge-resolution rate
  observed here (see next point), so it would need either a longer scheduled run or a faster
  bulk-load path for the edge fixture.

- **`block_bench` events/s vs `social_bench` events/s measure different paths — do not
  conflate them.** `block_bench` ingests raw single-node events in bulk via plain multi-row
  `INSERT` (no `MATCH`, no GQL edge resolution): 41,545 events/s. `social_bench` ingests via
  the GQL edge-program surface: node inserts are batched (1,000/statement), but each edge is
  still its own `MATCH (a:Person {_id: s}), (b:Person {_id: d}) INSERT (a)-[:KNOWS]->(b)`
  statement — the fixture batches up to 100 of these per tx/"program"
  (`varve_testkit::fixture::EDGE_PROGRAM_BATCH = 100`), giving 610 total txs (10 node batches +
  600 edge programs) for 70,000 events in 101.78 s → **688 events/s, 6 tx/s**. Despite the
  100-edges-per-tx batching, this is ~60× slower than `block_bench`'s bulk path, because the
  bottleneck here is **not** commit/fsync overhead — it is that each edge still pays two
  independent point `MATCH` resolutions (source + destination) inside the GQL engine, i.e.
  ~60,000 edges resolved in ~102 s ≈ 590 edges/s, or ~1.7 ms of resolution work per edge. This
  is v1's documented one-statement-per-edge mutation surface (multi-row edge `INSERT` bodies
  are roadmap, not v1) — it is an **end-to-end GQL ingest latency measurement**, not a bulk
  write-throughput measurement, and it must **not** be used to judge the ≥10k-write-ops/s
  (batched) target. That target is judged against `block_bench`'s 41,545 events/s bulk path
  above, which is met with margin.

- **`write_bench`'s 372 tx/s (local-fsync) is a third, narrower thing again**: one GQL
  statement = one row = one tx, deliberately unbatched, to measure the fsync-bound commit
  floor. It is healthy and is reported here as the "single-statement tx/s" figure, kept
  explicitly separate from both of the events/s figures above.

- **Point-lookup numbers differ across benches because they measure different things, not
  because of variance.** `social_bench`'s 0.38 ms warm point read is a same-process,
  already-open-`Db` query repeated 100× — the true "warm in-memory query latency" figure, and
  the one the <1 ms target is judged against. `block_bench` (7.55 ms) and `cache_bench` (7.75
  ms) instead report "warm" as *open+lookup after a fresh process restart with a
  cache/manifest that survived the restart* — i.e. they include a `Db::open()` (manifest read
  + log-tail replay) on every call, which dominates the measured time. Those two numbers are
  reported above for completeness (disk-cache-survives-restart behavior) but are not the
  figures used for the point-lookup verdict.

- **Server ≥5k tx/s on an object-store log: not measured this run.** No MinIO/S3 container or
  compose file was available in this worktree, and standing one up plus writing a dedicated
  concurrent-producer object-store-log tx/s bench was out of scope for this docs-only task —
  `docker ps` confirmed Docker itself works (0 containers running), so this is an availability
  gap in this session, not a fundamental blocker. The only real number in this direction is
  `write_bench`'s local-fsync group-commit path (372 tx/s, single local disk, single writer,
  one row/tx) — reported above for context but explicitly **not** a substitute for the
  object-store target: it is a different backend (local disk vs. S3-API object store), a
  different network topology (in-process vs. server + remote log), and an unbatched workload.
  Closing this requires: (1) a MinIO (or Garage) container reachable via `VARVE_S3_ENDPOINT`
  (`cache_bench` already supports pointing at one via env vars), (2) a dedicated concurrent
  multi-producer bench against `varved` writing through the object-store log with group commit
  enabled, and (3) enough sustained load (multiple concurrent clients) to characterize the
  group-commit window's effect on tx/s, since a single sequential writer (as in every bench in
  this report) cannot benefit from batching multiple in-flight commits together.

- **Scale-out ~linear to 4 nodes: measured, but single-box.** See the caveat under Scale-out
  above — the 152/282/530 QPS progression (1.86×/3.49× scaling) is real and encouraging, but
  all four query-node processes shared one machine's CPU and disk in this run. It is reported
  as "indicative," not a verified datacenter claim.

- **Machine load during measurement.** `uptime` showed a load average of 4.71 / 5.63 / 7.28
  (1/5/15-min) at the start of the run on a 14-core box (moderate background load, not a quiet
  dedicated rig), and the benchmark disk was at 90% capacity (95 GiB free). None of the runs
  above showed signs of pathological slowdown or hangs, but absolute latency numbers in this
  report should be read as "right order of magnitude on a laptop under everyday load," not
  lab-grade isolated measurements.
